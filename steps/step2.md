现在，我们正式进入 **第二步：数据准备——文档加载与文本切片（Data Ingestion & Chunking）**。

在这一步中，我们的目标是：**将一份长文档（PDF/TXT）导入系统，并将其合理地切分成适合大模型和向量数据库处理的小片段（Chunks）。**

---

### 💡 核心原理剖析（面试常考点）

在 RAG 系统中，“如何切分文档”是直接决定问答准确率的关键因素之一。

#### 1. 为什么我们不能把一整本书直接塞给大模型，而必须切片？
你可能会问：“现在的大模型（比如 GPT-4o、Claude 3.5）上下文窗口非常大（128k 甚至更多），直接把整份 PDF 丢给它不就行了吗？”
这在实际商业应用中是行不通的，原因有三点：
*   **成本极高 (Cost)：** 大模型是按 Token 收费的。如果你每次问答都把 10 万字的文档发过去，API 费用会呈指数级上升。
*   **响应速度慢 (Latency)：** 处理的数据量越大，大模型推理的速度越慢，用户体验极差。
*   **注意力迷失 (Lost in the Middle)：** 这是学术界公认的现象。当上下文过长时，大模型往往只能记住开头和结尾的信息，容易忽略文档中间的关键细节。

因此，**“只把与问题最相关的、几百字的代码/段落挑出来喂给大模型”** 才是最聪明、最经济的做法。

#### 2. 什么是切片策略 (Chunking Strategy)？
我们如何把一篇长文章切开？最简单的想法是：每 500 个字切一刀。但这会带来一个致命问题：
> *例子：* 假设一句话“张三是这个项目的负责人，他非常敬业”正好在第 500 个字这里被切断了：
> *   片段 A：“张三是这个项目的负责人，”
> *   片段 B：“他非常敬业。”
> 如果用户问“谁非常敬业？”，大模型拿到片段 B，根本不知道“他”指的是谁。

为了解决这个问题，LangChain 引入了两个核心设计：
*   **语义层级切分 (`RecursiveCharacterTextSplitter`)：**
    它不会生硬地按字符数切，而是会按照一个字符优先级列表 `["\n\n", "\n", " ", ""]`（双换行、单换行、空格、空字符）递归地寻找最合适的位置切。这样能尽量保证一个完整的段落、一句话被保留在同一个片段里。
*   **重叠区 (`chunk_overlap`)：**
    我们在相邻的片段之间保留一部分“重叠内容”（比如 50 个字）。这样即使某句话不幸被切在边缘，重叠区也能把前后的语义背景带过去，避免信息断裂。

#### 3. 编写文档解析与切片脚本
在项目文件夹下新建 `document_processor.py`，写入以下代码：

```python
from langchain_community.document_loaders import TextLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter

def process_document(file_path):
    # 1. 初始化加载器 (Loader)
    # TextLoader 用于读取纯文本文件。如果你用的是 PDF，可以使用 PyPDFLoader
    print(f"正在加载文档: {file_path}")
    loader = TextLoader(file_path, encoding="utf-8")
    documents = loader.load()
    
    # 2. 初始化切片器 (Text Splitter)
    # chunk_size: 每个片段的最大字符数
    # chunk_overlap: 相邻片段之间的重叠字符数
    text_splitter = RecursiveCharacterTextSplitter(
        chunk_size=150,       # 故意设小一点，方便我们观察切片效果
        chunk_overlap=30,      # 重叠区
        length_function=len,
    )
    
    # 3. 执行切片
    chunks = text_splitter.split_documents(documents)
    
    # 4. 打印切片结果以便分析
    print(f"\n文档加载成功！原始文档数: {len(documents)}")
    print(f"切片完成！共切分成 {len(chunks)} 个片段。")
    print("-" * 50)
    
    for i, chunk in enumerate(chunks):
        print(f"【片段 {i+1}】(字符数: {len(chunk.page_content)})")
        print(chunk.page_content)
        # 打印元数据，告诉你这个片段来自哪个文件的第几页/行
        print(f"元数据: {chunk.metadata}")
        print("-" * 50)

if __name__ == "__main__":
    process_document("sample_company_policy.txt")
```

### 遇到的问题
#### Q1. 这个运行结果是三段，按照第一章第二章第三章来分的，我没看到重复的区域，可能因为字符数最多是135，还没到chunk_size设置的值吗
A： 你的直觉和推理非常敏锐，**这就是底层切片器的真实运行机制！**
##### 深度原理解析：为什么没有出现重复区域？

`RecursiveCharacterTextSplitter` 的工作流程是这样的：
1. 它首先尝试用优先级最高的换行符 `\n\n`（即空行）去切分整篇文档。
2. 切完之后，它发现整篇文档被分成了三大块（第一章、第二章、第三章）。
3. 接着，它测量了这三大块的字符长度。正如你所发现的，每一章的字符数大约在 100~135 之间。
4. 此时，切片器看了一下你设置的 `chunk_size=150`。它心想：“既然这三块的长度都**已经小于** 150 了，那我就不需要再对它们进行更细的二次切分了。”
5. 既然不需要二次切分，那么**自然就不需要产生重叠区（Overlap）**。重叠区只在“某一块文本超长、被迫切成两半”时，为了连接语义才会产生。

> 💡 **小实验（可选）：** 如果你把 `chunk_size` 改成 `50`，把 `chunk_overlap` 改成 `15`，重新运行一次。你就会发现它在各章内部（比如第一条、第二条之间）进行了强行切分，并且切分点会出现 15 个字符的重复区域。
