# 📝 项目错题集：Bug 归档日志

## 📌 Bug 编号：001
*   **Bug 名称：** Windows 平台下运行 Chroma 向量库导致 Python 进程异常闪退（退出码 `0xC0000005`）
*   **发生阶段：** 第三步 - 构建向量数据库并进行数据持久化（Data Ingestion & Storage）
*   **技术栈：** Python 3.10+ / LangChain / ChromaDB / Windows 10/11

---

### 1. 现象描述 (Symptom)
在运行 `vector_store_demo.py` 进行向量化存储时，程序控制台输出：
```text
准备好 3 个文本片段，开始向量化...
正在计算向量并存入 Chroma 数据库...
```
随后，程序没有抛出任何标准的 Python 异常（如 `ValueError`、`TypeError`），而是直接无提示闪退。
在 PyCharm 或控制台中，显示进程以非零状态码退出：
```text
Process finished with exit code -1073741819 (0xC0000005)
```

---

### 2. 根本原因分析 (Root Cause)

#### ① 退出码 `0xC0000005` 是什么？
在 Windows 操作系统中，`0xC0000005` 代表 **`STATUS_ACCESS_VIOLATION`（内存访问违规 / 俗称：段错误）**。
这通常意味着程序试图读取或写入一段**它没有权限访问的物理内存地址**。在 Linux 操作系统中，这对应于 `SIGSEGV`。

#### ② 为什么 Python 的 `try...except` 无法捕获这个错误？
Python 的异常处理机制（如 `try...except`）只能捕获 Python 虚拟机（PyVM）层面的逻辑异常。
而当底层的 C/C++ 动态链接库（DLL）发生内存越界时，是 Windows 操作系统内核直接介入，出于安全保护机制，**强行终止（Kill）了整个 Python.exe 进程**。因为 Python 虚拟机本身都被系统杀掉了，所以根本来不及输出任何报错堆栈。

#### ③ 罪魁祸首：`ChromaDB` 与底层 C++ 库 `hnswlib` 的编译冲突
*   **底层依赖：** ChromaDB 为了实现极速的向量相似度检索，底层高度依赖一个名为 **`hnswlib`** 的 C++ 开源库（实现分层导航小世界算法）。
*   **硬件指令集与编译器冲突：** PyPI 上的预编译二进制包（Wheel）在 Windows 环境下，由于编译时使用的 MSVC 编译器版本与本地的 Windows C++ 运行时库（如 `msvcp140.dll`）存在兼容性差异；或者用户的 CPU 不完全支持 `hnswlib` 在编译时默认开启的某些 CPU 向量加速指令集（如 AVX/AVX2）。
*   **内存崩溃：** 当程序调用 `from_documents` 方法触发底层的 C++ 向量索引构建时，底层的机器码执行了非法的内存指令，直接触发 `0xC0000005` 内存冲突，系统强制关停进程。

---

### 3. 解决方案 (Engineering Solution)

#### 方案：无缝切换至工业级向量检索库 FAISS
*   **技术决策：** 考虑到本地开发环境的兼容性，选择将向量存储引擎由 `Chroma` 切换为 **`FAISS`（Facebook AI Similarity Search）**。
*   **为何选择 FAISS：** FAISS 是由 Meta（Facebook）开源并深度维护的工业级向量检索库。它的 Python 绑定包（`faiss-cpu`）在 Windows 下经过了更广泛的测试和编译优化，其动态 CPU 指令集探测非常稳定，不会发生此类兼容性内存冲突。
*   **代码解耦优势：** 得益于 LangChain 对向量数据库的高抽象设计（所有数据库都继承自统一的 `VectorStore` 类），本方案中**仅通过修改 2 行代码**（替换 `import` 和初始化方法），即完成了存储引擎的无缝替换，其余检索与上层逻辑无需任何修改。

## 📌 Bug 编号：002
*   **Bug 名称：** RAG 提示词过度约束（Prompt Over-Constraint）导致轻量大模型产生“防御性回答”（捧着正确答案说不知道）
*   **发生阶段：** 第四步 - RAG 完整问答链构建与调试阶段（LLM Generation）
*   **技术栈：** LangChain (LCEL) / 智谱 GLM-4-Flash / Prompt Engineering

---

### 1. 现象描述 (Symptom)
在 debug 模式下，控制台显示已经**精准检索**到了正确的文档片段：
`“第四条 餐饮补贴：加班超过晚上20:00的员工，可享受每日40元的加班餐饮补贴...”`
但当用户提问：`“我晚上加班到九点，公司会给报销晚饭钱吗？有金额限制吗？”` 时，大模型却直接触发了兜底逻辑，回答：
`“抱歉，根据公司规定守则，我没有找到相关信息。”`
而提问完全不相关的“五险一金”时，模型同样回答“没有找到相关信息”。

---

### 2. 根本原因分析 (Root Cause)

#### ① 强否定性提示词引发的“防御性回答 (Defensive Answering)”
在最初的 System Prompt 中，我们使用了极为严厉的否定性约束：
*   *“请**仅仅**基于下方提供的背景资料...”*
*   *“如果**完全没有**提到，请回答：‘我没有找到相关信息’...”*
*   *“**禁止编造**任何不存在的规定或事实！”*

像 `glm-4-flash` 这样的轻量级大模型（相比于 GPT-4o 或 GLM-4-Plus，它的参数量较小），在面对强否定性约束时，会表现出极度保守的“防御姿态”。为了绝对避免犯错（触发“禁止编造”的红线），它会倾向于放弃任何微小的逻辑推理。

#### ② 细微的“语义映射障碍 (Semantic Mapping Obstacle)”
用户的提问与参考资料之间存在两个微小的语义间距：
*   **时间尺度：** 提问是 `晚上九点（21:00）` ─── 资料是 `超过晚上20:00`。
*   **词汇映射：** 提问是 `晚饭钱` ─── 资料是 `餐饮补贴`。

对于大模型而言，`九点` 和 `20:00` 在字面上是不一样的。受到“禁止编造”的恐吓后，模型认为“将九点等同于超过20:00”或“将晚饭钱等同于餐饮补贴”存在极高的风险，于是它选择了最保险的做法——拒绝作答，直接给出兜底回复。

---

### 3. 解决方案 (Engineering Solution)

#### 方案：提示词结构优化（显式授权常识推理）
在不牺牲防幻觉底线的前提下，重构系统提示词，**主动、显式地向模型授权“常识推理”的边界**。
*   **引入【答题原则】：** 在 Prompt 中明确规定，允许并鼓励模型进行符合日常常识的合理语义映射（如“九点属于20:00之后”、“晚饭钱等同于餐饮补贴”）。
*   **软化否定语气：** 将原本生硬强烈的警告，替换为有逻辑、有边界的指引。
*   **成效：** 优化后，模型成功跨越了“九点”与“20:00”、“晚饭钱”与“餐饮补贴”的语义鸿沟，结合背景资料生成了准确的答案，同时对不相关的“五险一金”问题依然保持了严谨的拒绝态度。

---

### 4. 面试话术 / 项目总结 (Interview / Resume Value)
> **面试官提问：** “在 RAG 问答中，如果检索模块（Retrieval）工作正常，但生成模块（Generation）不给力，你会从哪些方面进行调优？”
>
> **完美回答：**
> “我遇到过一个典型案例：检索器已经成功把包含‘晚上 20:00 之后餐饮补贴 40 元’的片段喂给了大模型，但当用户问‘加班到九点能报销晚饭钱吗’时，轻量级模型依然回答‘不知道’。
> 经过诊断，我发现这是由于**提示词过度约束（Prompt Over-Constraint）**导致的。我们在写 System Prompt 时，为了防止幻觉，往往会使用‘绝对禁止编造’等强硬字眼。但这会导致轻量模型（如 glm-4-flash）采取‘防御性策略’，拒绝进行最基本的常识推理（如：九点属于20:00之后，晚饭钱就是餐饮补贴）。
> 针对这个痛点，我重构了提示词工程，在 Prompt 中建立了**‘常识推理授权区’**，明确告知模型哪些语义映射是允许的。这样既保留了系统对非文档知识的拒答能力（防幻觉），又大幅提升了对同义词、时间区间的泛化解答能力，使系统的可用性大幅提升。”

## 📌 Bug 编号：003
*   **Bug 名称：** 调用智谱 Embedding 接口时触发单次请求数组超限错误（Error Code 1214）
*   **发生阶段：** 第五步 - 多文档合并向量化阶段（Data Ingestion & Storage）
*   **技术栈：** LangChain / FAISS / ZhipuAI `embedding-3`

---

### 1. 现象描述 (Symptom)
当一次性上传多个文档（或单个超长文档）进行索引构建时，控制台抛出智谱 API 的请求失败异常：
`zhipuai.core._errors.APIRequestFailedError: Error code: 400, with error text {"error":{"code":"1214","message":"input数组最大不得超过64条"}}`
程序被迫中断，向量数据库构建失败。

---

### 2. 根本原因分析 (Root Cause)
*   **API 端的硬性约束：** 智谱 AI 等模型服务商为了限制服务器单次负载，对向量化接口（Embedding Endpoint）规定单次请求所包含的文本片段（Chunks）数组最大长度为 64。
*   **LangChain 默认行为机制：** 默认的 `FAISS.from_documents(all_chunks, embeddings)` 会尝试将分割后的所有文本切片打包成一个巨型数组，通过一次 HTTP 请求发送给大模型厂商。当切片总数大于 64 时，直接触发服务器的 1214 限制错误，请求被拒绝。

---

### 3. 解决方案 (Engineering Solution)
*   **技术决策：** 在向量化链路中引入**滑动窗口分批写入机制（Chunk Batching）**。
*   **具体实现：** 
  - 设定安全的单批次大小为 `batch_size = 50`（预留安全冗余）。
  - 使用 Python 切片循环，第一批次调用 `FAISS.from_documents(batch, embeddings)` 初始化数据库。
  - 剩余批次循环调用 `vector_store.add_documents(batch)`，增量且安全地将所有切片写入本地向量索引。
*   **成效：** 彻底攻克了海量文档及多文档上传时的接口瓶颈问题，系统健壮性达到了商用生产环境标准。

---

### 4. 面试话术 / 项目总结 (Interview / Resume Value)
> **面试官提问：** “如果用户上传了一个 10 万字的长文档，你的 RAG 系统在向量化时会面临什么挑战？你怎么解决？”
>
> **完美回答：**
> “这会直接导致云端大模型厂商的 API 请求超限。比如智谱 AI 的向量化接口限制单次请求文本量不得超过 64 条。
> 在项目中我通过引入**分批增量构建机制（Batching Ingestion）**解决了这个问题。我将切片集合以 50 条为安全阈值进行物理分包，首先创建向量数据库基底，后续批次通过 `add_documents` 增量追加。这种优雅的分批方案不仅保障了系统可以无上限地承载超长文档和多文档，也大幅优化了高并发下的网络请求健壮性。”


### 📌 Bug 004 | LangChain v1.0 主版本重构导致检索模块缺失

| 🏷️ 发生阶段 | 🛠️ 核心技术栈 | 🚨 严重程度 |
| :--- | :--- | :--- |
| 第五步 · 双路混合检索（FAISS + BM25）架构升级 | Python / LangChain v1.0.0+ / PyCharm | 🔴 High (编译性报错) |

#### 💥 现象描述 (Symptom)
在代码中引入集成检索器时，程序运行直接中断，终端抛出硬性导入异常：
```bash
ModuleNotFoundError: No module named 'langchain.retrievers'
```
同时，在 PyCharm 编辑器中，该导入语句即使运行成功也常伴有黄色或红色警告波浪线：
`在 '__init__.py' 中找不到引用 'retrievers'`

#### 🔍 根本原因 (Root Cause)
1. **主版本命名空间物理拆分（Package Splitting）：**
   LangChain 官方在发布 **v1.0.0+** 主版本后，对核心 `langchain` 库进行了彻底解耦与精简。原先属于 `langchain.retrievers`（包括核心组件 `EnsembleRetriever`）和 `langchain.chains` 的所有经典模块，已被**物理删除**并全部迁移至官方新维护的伴生库 **`langchain-classic`** 中。
2. **IDE 静态分析器对“延迟加载（Lazy Loading）”的虚警：**
   `langchain` 主包在底层初始化时大量采用了**动态元编程与按需加载**。PyCharm 的静态分析工具在不运行代码时，无法扫描出动态注册的 `retrievers` 模块，从而在编辑器中触发未解析引用警告。

#### 🚀 解决方案 (Engineering Solution)
1. **补全物理依赖包：** 在终端激活的 `env` 虚拟环境中安装新版伴生库：
   ```bash
   pip install langchain-classic
   ```
2. **重构物理导入路径：** 将代码中的导入语句更新为最新的 `langchain_classic` 命名空间：
   ```python
   # ❌ 旧版导入（在 v1.x 中已失效）：from langchain.retrievers.ensemble import EnsembleRetriever
   #  新版标准导入：
   from langchain_classic.retrievers import EnsembleRetriever
   ```
3. **添加编译器消除指令：** 在导入行上方加入 PyCharm 专用的编译器指令 `# noinspection`，消除 IDE 的静态检测虚警，保持项目目录清洁：
   ```python
   # noinspection PyUnresolvedReferences
   from langchain_classic.retrievers import EnsembleRetriever
   ```

### 📌 Bug 005 | Streamlit 页面重置导致参考文献闪退及 Document 对象非下标化异常

| 🏷️ 发生阶段 | 🛠️ 核心技术栈 | 🚨 严重程度 |
| :--- | :--- | :--- |
| 第五步 · 多会话历史持久化与参考文献实时渲染调试 | Streamlit (Rerun 机制) / LangChain `Document` | 🟡 Medium (交互异常与逻辑报错) |

#### 💥 现象描述 (Symptom)
系统在多会话历史调试和文献渲染中遭遇了双重异常：
1. **UI 闪退擦除：** AI 回答生成完毕后，下方的“参考文献出处”折叠面板瞬间展开，但随即**一闪而过并彻底消失**。
2. **物理取值报错：** 尝试渲染文件名时，控制台抛出硬性类型错误：
   ```text
   TypeError: 'Document' object is not subscriptable
   ```

#### 🔍 根本原因 (Root Cause)
1. **Streamlit 页面生命周期限制（Rerun）：**
   为了在用户发起首问时实现 ChatGPT 式的左侧边栏标题重命名与同步刷新，代码中调用了 `st.rerun()` 重跑页面。由于参考文献折叠面板是在 `st.chat_input` 临时作用域中“即时检索、即时渲染”的，`st.rerun()` 触发后该临时作用域的内存被系统擦除，而聊天历史记录（`st.session_state.messages`）此前只保存了回答纯文本，导致重新渲染后参考资料彻底丢失。
2. **对象类型与字典取值语法混淆：**
   双路检索返回的 `retrieved_docs` 中装的是原生的 **LangChain `Document` 类实例**。该对象只拥有 `.page_content` 与 `.metadata` 属性，**不支持 Python 字典的中括号下标取值语法（`[]`）**。在代码中使用 `doc['source']` 直接触发了底层的非下标化错误。

#### 🚀 解决方案 (Engineering Solution)
1. **规范原始对象取值：** 在大模型生成回答的在线检索部分，严格使用点号属性方式访问数据：
   ```python
   # 属性取值，解决 subscriptable 报错
   source_name = doc.metadata.get("source", "未知文件")
   page_text = doc.page_content
   ```
2. **参考文献历史持久化（Citations Persistence）：**
   * **数据层绑定：** 放弃在输入框执行块中进行临时渲染。在 AI 每次生成回答后，将文献的文件名和文本内容打包成字典列表，作为 `sources` 字段直接写入消息体中，随对话历史一同**序列化保存到本地 JSON 数据库**：
     ```python
     active_messages.append({
         "role": "assistant", 
         "content": ai_response,
         "sources": [{"source": doc.metadata.get("source", "未知"), "content": doc.page_content} for doc in retrieved_docs]
     })
     ```
   * **渲染层恢复：** 在主循环重新渲染历史聊天记录时，动态检测 AI 消息中是否含有 `sources` 字段。若有，则通过 `st.expander` 予以加载渲染（此时由于已序列化为 JSON，支持安全的下标取值）：
     ```python
     if msg["role"] == "assistant" and "sources" in msg:
         with st.expander("🔍 展开查看本次回答的参考文献出处"):
             for doc in msg["sources"]:
                 st.write(doc['source']) # 此时为 dict，下标访问安全
     ```
   * **成效：** 完美克服了 Rerun 导致的 UI 闪退 Bug；同时实现了功能升级：**所有历史会话在重启或切换时，每一条 AI 气泡都永久且独立绑定了它当时的参考文献引用出处！**
